5.2 Linux command line¶

The Linux command line interface (CLI) is a text-based interface. Often known as shell, terminal, console, prompt, command line, among other names, it may seem complex and confusing to use. However, the power and flexibility it offers means that its use can be essential when it comes to system administration.

This section explains the motivation for using the command line, teaches basic concepts and commands accompanied by some examples. It is assumed that the student has no prior knowledge.

A bit of history¶

During the early years of computing, UNIX, one of the first operating systems, was designed to run as a multi-user system on mainframe computers, with users connecting to it remotely through individual terminals. These terminals were quite basic by modern standards: only keyboard and screen, with no capability to run programs. Instead, they would simply send keystrokes to the server and display the data they received on the screen. There was no mouse, no sophisticated graphics, not even color. Everything was sent and received as text. Therefore, any program running on the mainframe had to produce text as output and accept text as input.

Compared to graphics, text consumes very few resources. Even on machines from the 1970s, which ran hundreds of terminals over extremely slow network connections, users could interact with programs quickly and efficiently. Very concise commands were created to reduce the number of keystrokes needed. This speed and efficiency is one of the reasons why this text interface is still widely used today.

When logging into a UNIX mainframe through a terminal, users had to perform administrative and file management tasks that are now done with a mouse and a couple of windows. Operations like creating, renaming, copying, or moving files on the disk were done with a text interface.

Each of these tasks required its own program or command: one to change directories (cd), another to list their contents (ls), a third to rename or move files (mv), etc. To coordinate the execution of each of these programs, the user would connect to a master program that could then be used to start any of the others. By wrapping the user's commands, this program, called a "shell", could provide common capabilities to any of them, such as the ability to pass data from one command to another, or use special wildcard characters to work with many files with similar names at once. Users could even write simple code, called "scripts", that could be used to automate long series of shell commands to facilitate complex tasks. The original UNIX shell program is "sh", but it has been expanded and replaced over the years, so on a modern Linux system, the most widespread use is of a shell called "bash".

Linux is a kind of descendant of UNIX. The core part of Linux is designed to behave similarly to a UNIX system, so most of the old shells and other text-based programs run smoothly. In theory, you could even connect one of those old terminals from the 1970s to a modern Linux system. But today, it is much more common to use a software terminal: the same old UNIX-style text interface, but running in a window alongside graphical interface.

Opening a terminal emulator¶

A terminal emulator is a computer program that simulates the operation of old physical terminals. In graphical interfaces, they are also known as terminal windows. From now on, we will refer to terminal emulators as terminals.

Linux distributions generally have a terminal launcher located in the same place as their other application launchers. It might be hidden in a submenu, but it is likely there somewhere. You can find the terminal by searching in the application launcher or search bar by typing the first few letters of "terminal", "console", "prompt", or "shell". Developers often configure the launcher with all the most common synonyms.

If there is no launcher, or you want a quicker way to open the terminal, most systems use the same default keyboard shortcut to launch it: Ctrl+Alt+T.

Regardless of how you start the terminal, you should end up with a rather dull window with some strange text at the top, like the image below. Depending on the Linux system, the colors might not be the same, and the text might say something different, but the general layout of a window with a large, mostly empty text area should be similar.

The terminal waits for command input, for example, you can type the command pwd in lowercase, and then press the Enter key to execute it.

The path of a directory (/home/YOUR_USER) will appear, followed by another copy of the prompt.

We need to understand a few basic concepts before we can comprehend exactly what the command does.

The Linux Shell¶

Shell programs are command interpreters that facilitate control of the operating system. They are not the operating system itself but form a layer that communicates the kernel with the interactive user. Upon logging into the system, each user runs a particular shell independent of any other user simultaneously using the system. This is indicated by the appearance of a character such as $, #, or > (there are others, and this is usually configurable) that depends on the type of shell launched. Shells are useful for executing one or more programs, showcasing the real multitasking capability of the operating system. Among the most popular shells are:

  • Bourne Shell: Written by Stephen Bourne, it is the most basic and oldest. It is invoked through the sh command, after which the system prompt $ should appear.
  • C Shell: Subsequent to the Bourne Shell and oriented towards C programmers. It is executed by typing the csh command, resulting in the % or > character. An improved version of this is tcsh, which allows file and directory name completion using the tab key and command line editing. This latter version is usually available in all Linux distributions.
  • Korn Shell: An extension of the Bourne Shell. It is launched by executing the ksh command, resulting in the $ prompt.
  • GNU Bourne-Again SHell (bash): As its name suggests, it is an extension of the Bourne Shell and adds features such as file name completion (similar to tcsh) and listing files that start with a specific sequence of characters by simply pressing the tab key. It is invoked using the bash command.

To check the list of interpreters available on the system, you can run cat /etc/shells.

$ cat /etc/shells 

# /etc/shells: valid login shells
/bin/sh
/bin/bash
/usr/bin/bash
/bin/rbash
/usr/bin/rbash
/bin/dash
/usr/bin/dash

To find out the default shell configured on the system, you can run echo $SHELL

$ echo $SHELL
/bin/bash

Although the $SHELL variable does not tell us which interpreter we are currently using, to find out which shell we are using, we can use ps -p $$.

Prompt¶

When opening the terminal emulator, a line of text appears in front of the blinking cursor (user@host:~$ _) known as the prompt, indicating that the terminal is ready to accept commands. The prompt is configurable, but many systems use the default format seen in the example:

  • Username (user). The user we are currently working with.
  • Hostname (host). The name of the computer to which the terminal is connected. This might seem obvious, but if we connect to another computer remotely, we might forget where we are working. This way, we can quickly see it.
  • At symbol (@). You might recognize the symbol from email addresses. For example, if we have "admin@PC01," it would be read as "admin at PC01," meaning that the user "admin" is connected to the computer "PC01"
  • Colon (:). Separates the hostname from the current working directory.
  • Current working directory. In the example, ~ is an abbreviation for the user's home directory.
  • Dollar sign ($) or hash (#). Depending on the user's permissions, the prompt will show a dollar sign for regular users and a hash for the superuser (root).
  • Cursor. Indicates where we can enter new text, which might be a blinking bar, a square block, or an underscore.

Commands¶

A command is a program with a specific functionality. By typing a text entry at the prompt, we are requesting the terminal to execute a command. Whether it succeeds or not, once the command execution is complete, the terminal will display the prompt again and wait for a new command.

Commands can be accompanied by options and arguments that modify their behavior.

For example, in the command ls -a /usr

  • ls is the command responsible for displaying the contents of a directory.
  • -a is an option that specifies the output format
  • /usr is an argument that indicates the directory we want to display.

Options and arguments are separated by spaces.

Options are usually preceded by a hyphen (-) and can be combined. For example, ls -la /usr is equivalent to ls -l -a /usr.

Basic Navigation¶

File System Organization¶

Like MS Windows, files on a Linux system are organized in what is called a hierarchical directory structure. This means they are organized in a tree-like pattern of directories, which can contain files and subdirectories. The first directory in the file system is called the root directory. The root directory contains files and subdirectories, which contain more files and subdirectories, and so on.

Most graphical environments include a file manager program used to view and manipulate the contents of the file system.

One important difference between MS Windows and Linux operating systems is that Linux does not use the concept of drive letters. While MS Windows drive letters divide the file system into a series of different trees (one for each volume), Linux always has a single tree. Different storage devices may be different branches of the tree, but there is always a single tree.

'pwd' command¶

Since the command line interface cannot provide images of the file system structure, we need a different way to represent it. To do this, think of the file system tree as a maze and that we are inside it. At any given moment, we are located in a single directory. Within that directory, we can see its files, the path to its parent directory, and the paths to the subdirectories of the directory we are in.

The directory we are in is called the working directory. To display the name of the current working directory, we use the pwd command.

user@DEB11:~$ pwd
/home/user

When we first log into the system, the working directory is set to our home directory. This is where we put our files. On most systems, the home directory will be called /home/username, (although it could be changed by the administrator).

To list the files in the working directory, we use the ls command.

user@DEB11:~$ ls
Downloads  Desktop  Templates  Videos
Documents  Pictures  Music  Public

'cd' command¶

To change the current working directory, we use the cd command. To do this, type cd followed by the path of the desired working directory. A path name is the route we take along the branches of the tree to reach the directory we want. Path names can be specified in two different ways: absolute paths or relative paths. Let's first look at absolute path names.

An absolute path name starts with the root directory and follows the tree branch by branch until the path to the desired directory or file is complete. For example, there is a directory on your system where most programs are installed. The path name of the directory is /usr/bin. This means that from the root directory (represented by the initial slash / in the path name) there is a directory called "usr" that contains a directory called "bin".

user@DEB11:~$ cd /usr/bin

We can see that we have changed the current working directory to /usr/bin and it is full of files. Notice how the prompt has changed to user@DEB11:/usr/bin$.

Where an absolute path name starts from the root directory and leads to its destination, a relative path name starts from the current working directory. To do this, it uses a couple of special notations to represent relative positions in the file system tree. These special notations are "." (colon) and ".." (two colons).

The "." (colon) refers to the current working directory and the notation ".." (two colons) refers to the previous or parent directory. Here is how it works. Let's change the working directory to /usr/bin again:

user@DEB11:~$ cd /usr/bin
user@DEB11:/usr/bin$ pwd
/usr/bin

Alright, now let's say we want to change the working directory to the parent of /usr/bin, which is /usr. We could do this in two different ways. First, with an absolute path:

user@DEB11:/usr/bin$ cd /usr
user@DEB11:/usr$ pwd
/usr
Or with a relative path to the current working directory: 
user@DEB11:/usr/bin$ cd ..
user@DEB11:/usr$ pwd
/usr

Two different methods with identical results. Which one should we use? The one we consider requires fewer keystrokes.

Likewise, we can change the working directory from /usr to /usr/bin in two different ways. First, using an absolute path:

user@DEB11:/usr$ cd /usr/bin
user@DEB11:/usr/bin$ pwd
/usr/bin

O con la ruta relativa al directorio de trabajo actual:

user@DEB11:/usr$ cd ./bin
user@DEB11:/usr/bin$ pwd
/usr/bin
In most cases, we can omit the "./", it is implied: 
user@DEB11:/usr$ cd bin
user@DEB11:/usr/bin$ pwd
/usr/bin

In general, if we do not specify a path name, the working directory will be assumed. However, there are exceptions that we will see later.

Some shortcuts¶

If we type cd without arguments, cd will change to our user directory /home/user_name.

A related shortcut is to type cd ~user_name, in this case, cd will change the working directory to the specified user's home directory.

Typing cd - changes the working directory to the previous one.

The importance of UPPERCASE and lowercase¶

Care must be taken with the use of uppercase and lowercase letters when typing a command. Typing PWD will produce an error, which if we do not pay attention, we might think it executed correctly, so it is important to read the command output.

This also applies to file and directory names. For example, "File1", "file1" and "FILE1" would be three different files.

Requesting help¶

Built-in help --help¶

Most commands have a built-in help system that can be accessed by typing the command name followed by the --help option or -h. For example, to access the help system for the ls command, type ls --help.

user@DEB11:/usr/bin$ ls --help

Bash built-in help help¶

The help command provides information about the built-in commands of the Bash shell. To access the help system for a built-in command, type help followed by the command name. For example, to access the help system for the cd command, type help cd.

user@DEB11:/usr/bin$ help cd

If you are not sure of the command name, you can use the help command to search for commands that contain a specific keyword.

user@DEB11:/usr/bin$ help

Manual pages man¶

Most commands have a manual page that provides detailed information about the command, including its options and arguments. To access the manual page, type man followed by the command name. For example, to access the manual page for the ls command, type man ls.

user@DEB11:/usr/bin$ man ls

The manual page is displayed one page at a time. To move forward one page, press the space key. To move back one page, press the b key. To exit the manual page, press the q key.

If you are not sure of the command name, you can use the apropos command to search for commands that contain a specific keyword. For example, to search for commands related to the keyword password, type apropos password.

user@DEB11:/usr/bin$ apropos password

Looking around the system¶

Now that we know how to move from one working directory to another, we can take a tour of the Linux system. Next, we need to know some tools that will be useful to understand the contents of directories.

  • ls list files and directories
  • cat display the contents of a text file
  • less navigate through the contents of text files
  • file determine the content of a file

The ls command is used to list the contents of a directory. It is probably the most used Linux command. It can be used in several different ways. Here are some examples:

'cat' command¶

Was created to concatenate files, but it also allows us to view text files. This is very useful since many of the files used to control and configure Linux are human-readable. For example, to display the contents of a file called file.txt, we would type:

cat file.txt

What is a "text file"?

There are many ways to represent information on a computer. All methods involve defining a relationship between the information and some numbers that will be used to represent it. Computers, after all, only understand numbers, and all data is converted into numerical representation.

Some of these representation systems are very complex (such as compressed multimedia files), while others are quite simple. One of the earliest and simplest is called ASCII text. ASCII stands for American Standard Code for Information Interchange. This is a simple encoding scheme that was first used in teletype machines to map keyboard characters to numbers.

Text is a simple one-to-one mapping of characters to numbers. It is very compact. Fifty characters of text translate into fifty bytes of data. Throughout a Linux system, many files are stored in text format, and there are many Linux tools that work with text files. Even MS Windows systems recognize the importance of this format. The well-known program NOTEPAD.EXE is a plain ASCII text file editor.

Manipulating Files and Directories¶

To introduce basic file and directory manipulation, we will look at the following commands:

  • cp - copy files and directories
  • mv - move or rename files and directories
  • rm - delete files and directories
  • mkdir - create directories

These four commands are among the most used in Linux. They are the basic commands for manipulating both files and directories.

Some of the tasks performed by these commands are easier to do with a graphical file manager. With a file manager, you can drag and drop a file from one directory to another, cut and paste files, delete files, etc. So, why use these old command-line programs?

The answer is power and flexibility. While it is easy to perform simple file manipulations with a graphical file manager, complex tasks can be easier with command-line programs.

For example, how do you copy all HTML files from one directory to another, but only copy files that do not exist in the destination directory or are newer than the versions in the destination directory? Quite difficult with a file manager. Quite easy with the command line:

$ cp -u *.html destination

Wildcards¶

Before starting with the commands, let's first look at a feature of the shell that makes these commands so powerful. Since the shell uses filenames a lot, it provides special characters to help you quickly specify groups of filenames. These special characters are called wildcards. Wildcard characters allow you to select filenames based on character patterns. The following table lists the wildcard characters and what they select:

Wildcard Meaning
* Matches any set of characters
? Matches any single character
[chars] Matches any character that is a member of the set characters. The set of characters can also be expressed as a POSIX character class, such as one of the following:
[:alnum:] Alphanumeric characters, same as [a-zA-Z0-9]
[:alpha:] Alphabetic characters, same as [a-zA-Z]
[:digit:] Numeric characters, same as [0-9]
[:upper:] Uppercase alphabetic characters, same as [A-Z]
[:lower:] Lowercase alphabetic characters, same as [a-z]
[!chars] Matches any character that is not a member of the set characters

Using wildcards, it is possible to construct very sophisticated selection criteria for filenames. Here are some examples of patterns and what they match:

  • * All filenames
  • g* All filenames that start with the character "g"
  • b*.txt All filenames that start with the character "b" and end with the characters ".txt"
  • Data??? Any filename that starts with the characters "Data" followed by exactly 3 more characters
  • [abc]* Any filename that starts with "a" or "b" or "c" followed by any other character
  • [[:upper:]]* Any filename that starts with an uppercase letter. This is an example of a character class.
  • BACKUP[[:digit:]][[:digit:]] Another example of character classes. This pattern matches any filename that starts with the characters "BACKUP" followed by exactly two numbers.
  • *[![:lower:]] Any filename that does not end with a lowercase letter.

We can use wildcards with any command that accepts filename arguments.

'cp' command¶

The cp program copies files and directories.

The most basic usage is to copy a file.

$ cp source_file destination_file

It can also be used to copy multiple files (and/or directories) to another directory:

$ cp file... destination_directory

About notation (...)

In the documentation of a command, ... means that an element can be repeated one or more times.

Examples of using the cp command:

  • cp file1 file2

    • Copies the contents of file1 to file2.
      • If file2 does not exist, it is created.
      • Otherwise, file2 is overwritten without warning.

  • cp -i file1 file2

    • Like the previous example, but the -i (interactive) option.
    • Prompts the user for confirmation before overwriting file2 if it exists.

  • cp file1 dir1

    • Copies the contents of file1 to a new file named file1 inside the directory dir1.

  • cp -R dir1 dir2

    • Uses -R option (recursive).
    • Copies the contents of the directory dir1 to the directory dir2.
      • If dir2 does not exist, it is created.
      • If dir2 exists, a directory named dir1 is created inside dir2.

'mv' command¶

The mv command moves or renames files and directories depending on how it is used. It can also move one or more files to a different directory or rename a file or directory.

To rename a file, use:

$ mv original_name new_name

To move files (and/or directories) to a different directory:

$ mv file... destination_directory

Examples of using the mv command:

  • mv file1 file2
    • If file2 does not exist, then file1 is renamed to file2.
    • If file2 exists, its contents are replaced without warning.
  • mv -i file1 file2
    • Like the previous example, but with the -i (interactive) option.ç
    • If file2 exists, it asks for user confirmation before overwriting its contents.
  • mv file1 file2 dir1
    • The files file1 and file2 are moved to the directory dir1.
  • mv dir1 dir2
    • If dir2 does not exist, dir1 is renamed to dir2.
    • If dir2 exists, the directory dir1 is moved inside dir2.

'rm' command¶

The rm (remove) command deletes files and directories.

$ rm file...

Using the -r option removes directories and their contents recursively.

$ rm -r directory...

Examples of using the rm command:

  • rm file1
    • Deletes file1
  • rm file1 file2
    • Deletes file1 and file2
  • rm -i file1 file2
    • Like the previous example, but the -i (interactive) option.
    • Prompts the user for confirmation before deleting each file.
  • rm -r dir1 dir2
    • The directories dir1 and dir2 are deleted along with all their contents.

Be careful with rm

Files deleted using the rm command cannot be recovered, they do not go to the trash. If we are not careful, we can cause damage to the system, especially when using wildcards.

Before using rm with wildcards, it is advisable: to construct the command using ls to see which files are selected with the wildcards. After verifying that the files are correct, we can replace ls with rm.

'mkdir' command¶

The mkdir command is used to create directories:

$ mkdir directory...

Examples of using the mkdir command:

  • mkdir dir1
    • Creates a directory called dir1
  • mkdir dir1 dir2
    • Creates directories dir1 and dir2
  • mkdir -p dir1/dir2/dir3
    • Creates the directory dir1 and inside it the directory dir2 and inside it the directory dir3.
    • The -p option creates all the directories in the path that do not exist.

Using commands with wildcards¶

The commands we have seen can use multiple files and directories as arguments, and we can use wildcards to specify them.

Examples of using commands with wildcards:

  • cp *.txt Documents
    • Copies all files with the .txt extension from the current directory to an existing directory called Documents
  • mv dir1 ../*.bak dir2
    • Moves the subdirectory dir1 and all files with the .bak extension from the parent directory to a directory called dir2
  • rm *.jpg
    • Deletes all files in the current directory that end with the .jpg extension.

There are four types of commands:

  1. Executable programs such as the files in /usr/bin. These programs can be compiled binaries or programs written in interpreted scripting languages.

  3. Shell functions. These are small scripts incorporated into an environment. They are studied in shell script programming.
  4. Aliases. Commands that we can define, composed of other commands.

The type command is built into the shell and shows the type of a given command.

Where 'command' is the name of the command we want to examine.

cd is a shell builtin

Input and Output Redirection¶

Input and output redirection is a powerful feature used by command-line programs. Many commands like ls print their output to the screen. However, this doesn't have to be the case. Using special notations, we can redirect the output of many commands to files, devices, and even to the input of other commands.

In all variants of Unix, there are three standard streams that connect the input and output (I/O) of a command/application with the terminal when executed. They are as follows:

  • STDIN (0) - Standard input, data entered into the program, by default through parameters entered as arguments.
  • STDOUT (1) - Standard output, data returned by the program, by default to the terminal.
  • STDERR (2) - Standard error, for error messages, also by default to the terminal.

Each channel has a unique number associated with it to identify it. This is known as a file descriptor.

Standard Output '>', '>>'¶

Command-line programs display their results to a device called standard output or STDOUT. By default, shows its content to the same terminal. To redirect the standard output to a file, use the ">" character as follows:

$ ls > file_list.txt

In this example, the ls command is executed, and the results are written to a file called file_list.txt. Since the output of ls was redirected to the file, no results will appear on the screen.

Each time the above command is repeated, file_list.txt is overwritten from the beginning with the output of the ls command.

To append the new results to the file, use ">>" as follows:

$ ls >> file_list.txt

The results are added to the end of the file, making the file longer each time the command is repeated. If the file does not exist when attempting to append the redirected output, the file will be created.

Standard Input '<'¶

Many commands can accept input from a device called standard input or STDIN. By default, standard input gets its content from the keyboard, but like standard output, it can be redirected. To redirect standard input from a file instead of the keyboard, use the "<" character as follows:

$ sort < file_list.txt

In the example above, we used the sort command to process the content of file_list.txt. The results are displayed on the screen since the standard output was not redirected. We could redirect the standard output to another file as follows:

$ sort < file_list.txt > sorted_file_list.txt

As we can see, a command can have both its input and output redirected. Note that the order of the redirection does not matter. The only requirement is that the redirection operators (the "<" and ">") must appear after the command's other options and arguments.

Pipes '|'¶

The most useful and powerful thing we can do with I/O redirection is to connect multiple commands together to form what is called pipelines. With pipelines, the standard output of one command is fed into the standard input of another. Here is a very useful example:

$ ls -l | less

In this example, the output of the ls command is fed into less. Using this "| less" trick, we can make any command have interactive output where we can paginate and search through the output.

By connecting commands together, we can make the most out of the command line. Here are some examples:

  • ls -lt | head Displays the 10 newest files in the current directory.
  • du | sort -nr Displays a list of directories and how much space they consume, sorted from largest to smallest.
  • find . -type f -print | wc -l Displays the total number of files in the current working directory including all its subdirectories.

Filters¶

A type of program frequently used with pipelines is called a filter. Filters take standard input, perform an operation on it, and send the results to standard output. In this way, they can be combined to process information in very powerful ways. Here are some of the most common programs that can act as filters:

  • sort Sorts the standard input and displays the sorted result to the standard output.
  • uniq Given a sorted stream of data from the standard input, removes duplicate lines of data, ensuring each line is unique.
  • head Outputs the first lines of its input. Useful for getting the header of a file.
  • tail Displays the last lines of the input. Useful for things like getting the most recent entries from a log file.
  • grep Examines each line of data it receives from the standard input and displays all lines that contain a specific pattern of characters.

Other useful filters include:

  • tr Translates characters. It can be used for tasks like uppercase/lowercase conversions or changing line-ending characters from one type to another (e.g., converting DOS text files to Unix-style text files).
  • sed Stream editor. It can perform more sophisticated text translations than tr.
  • awk A full-fledged programming language designed for building filters. Extremely powerful.
  • fmt Reads text from the standard input and then outputs formatted text to the standard output.
  • pr Takes text from the standard input and splits the data into pages with page breaks, headers, and footers in preparation for printing.

Usage examples:

Printing from the command line. Linux provides a program called lpr that accepts standard input and sends it to the printer. It is often used with pipes and filters. Here are a couple of examples:

$ cat poorly_formatted_report.txt | fmt | pr | lpr

The example follows these steps:

  • cat reads the file and sends it to standard output, which is piped to the standard input of fmt.
  • fmt formats the text into neatly arranged paragraphs and sends it to standard output, which is piped to the standard input of pr.
  • pr neatly divides the text into pages and sends it to standard output, which is piped to the standard input of lpr.
  • lpr takes its standard input and sends it to the printer.
$ cat unsorted_list_with_duplicates.txt | sort | uniq | pr | lpr

The example starts with an unsorted list of data with duplicate entries. First, cat sends the list to sort, which sorts it and sends it to uniq, which removes the duplicates. Then pr and lpr are used to paginate and print the list.

Viewing the contents of tar files. Often, you will receive tar files compressed with gzip. You can use the following command to view the contents of such a file on a Linux system:

$ tar tzvf archive_name.tar.gz | less

Standard Error Output '2>'¶

The standard error output is where the command sends error data while performing its task. This data stream is also printed to the terminal but is not sent to the standard input (STDIN) of a consuming command.

$ ls -l valid.txt invalid.txt
ls: cannot access 'invalid.txt': No such file or directory
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt

In the example, an error appears in the terminal indicating that invalid.txt does not exist, followed by the data of the second file valid.txt which does exist.

If we redirect the standard output of the command to a file, we get the following:

$ ls -l valid.txt invalid.txt > output.txt
ls: cannot access 'invalid.txt': No such file or directory

$ cat output.txt
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt

We can see that the error still appears in the terminal, and the file stores the output of the file that did not produce an error.

We could store the standard error output by redirecting the output with "2>":

$ ls -l valid.txt invalid.txt 2> error.txt
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt

$ cat error.txt
ls: cannot access 'invalid.txt': No such file or directory

Often, the standard error output is discarded by redirecting it to the /dev/null device, so it is neither displayed nor stored anywhere.

$ ls -l valid.txt invalid.txt 2> /dev/null
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt

If we want to redirect both outputs, each to a different file, we do the following:

$ ls -l valid.txt invalid.txt > output.txt 2> error.txt

$ cat output.txt
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt

$ cat error.txt
ls: cannot access 'invalid.txt': No such file or directory

Finally, if we want to redirect the standard error output to the standard output to store both outputs in the same file, we do the following:

$ ls -l valid.txt invalid.txt > output.txt 2>&1

$ cat output.txt
-rw-rw-r-- 1 user user 11 Mar 30 12:42 valid.txt
ls: cannot access 'invalid.txt': No such file or directory

Permissions¶

Linux permissions indicate three things you can do with a file or directory: read, write, and execute. Each permission is represented by a letter.

  • r read - you can view the content of the file.
  • w write - you can change the content of the file.
  • x execute - you can run the file if it is a program or script.

The same set of permissions can be used for directories, but they have a slightly different behavior.

  • r - you can read the contents of the directory, i.e., perform an ls.
  • w - you can write to the directory, i.e., create files and directories.
  • x - you can enter the directory, i.e., execute cd.

Permissions are divided into three categories:

  • owner - the single user who owns the file. (usually the person who created the file, but ownership can be transferred to another user)
  • group - each file belongs to a single group.
  • others - any other user who is not the owner or part of the group.

Checking Permissions¶

With ls -l, we can see the details of the files, including their permissions.

$ ls -l archivo.txt
-rwxr----x 1 alumno1 students 12 Mar 29 09:23 archivo.txt

In the example above, the first 10 characters of the output are what we look at to identify the permissions -rwxr----x.

  • The first character identifies the type of file. If it is a dash (-), it is a regular file. If it is a d, it is a directory.
  • The next three characters represent the owner's permissions. A letter represents the presence of a permission, and a dash (-) represents the absence of a permission. In this example, the owner has all permissions (read, write, and execute).
  • The next three characters represent the group's permissions. In this example, the group has read permission but no write or execute permissions. Note that the order of permissions is always read, then write, then execute.
  • Finally, the last three characters represent the permissions for others (or everyone else). In this example, they have execute permission and nothing else.
  • Additionally, we can see that the file belongs to the user alumno1 and the group students.

Changing Permissions 'chmod'¶

To change the permissions of a file or directory, use a command called chmod.

chmod [permissions] [path]

The permissions parameter can be passed in two different ways: numeric notation and symbolic notation.

Example of changing permissions using numeric notation:

$ chmod 750 file.txt
$ ls -l file.txt
-rwxr-x--- 1 student1 students 12 Mar 29 09:23 file.txt

Example of changing permissions using symbolic notation, in this case, removing the execute permission:

$ chmod -x file.txt
$ ls -l file.txt
-rw-r----- 1 student1 students 12 Mar 29 09:23 file.txt

Symbolic notation is more flexible as it allows adding or removing permissions without modifying the rest.

We can specify where the permission applies using the following arguments:

  • u user owner
  • g group
  • o others
  • a all

Additionally, we can indicate the operation with the following symbols:

  • + adds the permission
  • - removes the permission
  • = sets the permission with the specified values

Some examples:

  • +x adds execute permissions to all, equivalent to using a+x
  • u+x adds execute permission to the user owner
  • g-x removes the execute permission for the group
  • o-rw removes read and write permissions for others
  • go=rw sets the permissions for the group and others to read and write but not execute
  • u+x,go=rx adds execute permission to the owner and sets the permissions for the group and others to read and execute but not write

Process Management¶

Linux is generally a very stable system. However, from time to time, performance may degrade, or it may be necessary to adjust the system's operation to better suit our needs. In this section, we will take a brief look at how we can manage programs, or processes, on a Linux system.

A program is a series of instructions that tell the computer what to do. When we run a program, those instructions are copied into memory, and space is allocated for variables and everything needed to manage its execution. This running instance of a program is called a process, and we can manage it.

Foreground and Background Jobs¶

When we run a program or command, it usually runs in the foreground. Most of them execute in a fraction of a second. We might want to start a process that takes a long time and runs without our intervention (processing a very large text file or compiling a program, for example). What we can do is run the program in the background and continue working. We will demonstrate this with a program called sleep. All sleep does is wait for a specified number of seconds and then exit. We can also use a command called jobs that lists the background jobs that are running.

$ sleep 60

This command would block the command line for 60 seconds.

If we want to cancel the execution, we can press the Ctrl + C key combination. If we press Ctrl + Z, the process stops and is sent to the background.

To run a command in the background, we add an ampersand & at the end of the command.

$ sleep 60 &

To check the processes running in the background, we use the jobs command.

$ sleep 60 &
[1] 16656
$ jobs
[1]+  Running                 sleep 60 &

To bring a process to the foreground and continue its execution, we use the fg (foreground) command and the job number.

$ fg 1
sleep 60

If we have sent a process to the background with Ctrl + Z, it will appear stopped in jobs.

$ sleep 60
^Z
[1]+  Stopped                 sleep 60
$ jobs
[1]+  Stopped                 sleep 60

To resume the execution of the process but keep it in the background, we use the bg (background) command followed by the job number.

$ bg 1
[1]+ sleep 60 &
$ jobs
[1]+  Running                 sleep 60 &

You can view all running processes on the system, along with their command line arguments, as well as see them in a tree format, select multiple processes, and act on all of them at once.

'ps' command¶

Another program to view processes is ps. In its normal usage, it will only show the processes running in the current terminal (which are usually not many).

$ ps
  PID TTY          TIME CMD
  46141 pts/0    00:00:00 bash
  46221 pts/0    00:00:00 ps

If we add the aux argument, it will display a complete view of the system processes.

$ ps aux
USER   PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root     1  0.0  0.9  98464  9516 ?        Ss   12:18   0:00 /sbin/init
root     2  0.0  0.0      0     0 ?        S    12:18   0:00 [kthreadd]
root     3  0.0  0.0      0     0 ?        I<   12:18   0:00 [rcu_gp]
root     4  0.0  0.0      0     0 ?        I<   12:18   0:00 [rcu_par_gp]
...

It has a much more extensive output, so it is often piped to grep to filter the results.

'kill' command¶

The kill command allows you to stop a process by specifying its ID. In reality, kill is used to send a signal to a process, but if no options are specified, it sends the termination signal -TERM.

With kill -l, you can see a list of available signals, the most important ones are:

  • SIGTERM (15) - Software termination signal (sent by kill by default).
  • SIGKILL (9) - If a process receives this signal, it must exit immediately and will not perform any cleanup operations.
  • SIGSTOP (19) - Issued to stop a process.
  • SIGCONT (18) - Continue the execution of a stopped process.
  • SIGHUP (1) - Issued when the associated terminal is closed.
  • SIGINT (2) - Issued if the user sends an interrupt signal (Ctrl + C).
  • SIGQUIT (3) - Issued if the user sends a quit signal (Ctrl + D).

The signals have an associated standard numeric code for different processor architectures.

Example of using kill:

$ sleep 5000&
[1] 18456
$ ps -a
  PID TTY          TIME CMD
  18456 pts/2    00:00:00 sleep
  18457 pts/2    00:00:00 ps
$ kill 18456
$ ps -a
  PID TTY          TIME CMD
  18457 pts/2    00:00:00 ps

In the example, sleep is executed to run for 5000 seconds, but it is launched in the background with &. We can see that some numbers appear, [1] 18456, where the number in brackets indicates the background job, and the other number is the process ID. With ps -a, we see the active processes in the current terminal along with their PID (Process ID). By executing kill along with the PID, we are sending the TERM (15) signal, asking the process to terminate its execution in an orderly manner.

If an application or command is stuck and we want to forcibly terminate its execution, we will use the KILL (9) signal.

$ kill -9 18456

You can also specify the signal as text:

$ kill -KILL 18456

This has the same effect as the previous example but forces the application to close. For example, if we have a text editor open with unsaved changes, generally, the text editor, when requesting an orderly shutdown, signal TERM (15), will display an interactive message or dialog asking the user if they want to save the changes. If we use the KILL (9) signal, this message will not appear, and the changes will be lost.

Links¶

In Linux, everything is treated as a file. However, there is a special type of file called a link, and there are two types of links:

  • Symbolic links: Also known as soft links, they point to the path of another file. If the file to which the link points (called the target) is deleted, the link will still exist, but it will no longer work.

  • Hard links: Also known as physical links, they act as a second name for the original file. They are not duplicates but rather an additional entry in the file system that points to the same location (inode) on the disk.

An inode is a data structure that stores attributes for an object (such as a file or directory) in a file system. These attributes include permissions, ownership, and the disk blocks where the object's data is stored. It can be understood as an entry in an index, hence the name, which comes from "index node".

Hard Links¶

The command to create a hard link is ln. Its basic syntax is as follows:

ln TARGET LINK_NAME

The TARGET must exist; it is the file to which the link will point. If the target is not in the current directory, or if you want to create the link elsewhere, you must specify the full path. For example:

$ ln document.txt /home/user/Documents/hardLink.txt

This will create a file named hardLink.txt in the /home/user/Documents/ directory linked to the document.txt file located in the current directory.

If the LINK_NAME parameter is omitted, a link with the same name as the target will be created in the current directory.

Hard links are entries in the file system that have different names but point to the same data on the disk. All these names can be used to reference a single file. If the content of one of the hard links is changed, the content of all other links pointing to that file changes, as all these links point to the same disk data. If one of the links is deleted, the other links will still work.

This happens because when a file is deleted, the data is not removed from the disk. The system simply removes the entry in the file system table that points to the inode corresponding to the data on the disk. But if you have a second entry pointing to the same inode, you can still access the data.

This can be verified using the -i parameter of ls. As an example, let's consider the following directory contents:

$ ls -li
total 224
4513695 -rw-r--r-- 2 user student 113402 Mar 24 11:31 hardLink
4513695 -rw-r--r-- 2 user student 113402 Mar 24 11:31 target.pdf

The number that appears before the permissions is the inode number. The files hardLink and target.pdf have the same number (4513695), indicating that one is a hard link to the other.

But which one is the original and which is the link? In practice, it is impossible to tell, as they are essentially the same file.

It should be noted that each hard link pointing to a file increases the file's link count. This is the number that appears after the permissions in the ls -l output. By default, each file has a link count of 1 (directories have a count of 2), and each hard link increases the count by one. This explains the link count of 2 in the files in the example above.

Unlike symbolic links, you can only create hard links to files, and both the link and the target must reside on the same filesystem.

Since hard links are treated as normal files, they can be deleted with rm and renamed or moved around the filesystem with mv. And because a hard link points to the same inode as the target, it can be moved freely without breaking the link.

Symbolic Links¶

The command used to create a symbolic link is also ln, but with the -s parameter added. Its basic syntax is:

ln -s TARGET LINK_NAME

Given the following example:

$ ln -s target.txt /home/user/Documents/symbolicLink

This will create a file named symbolicLink in the /home/user/Documents/ directory, pointing to the target.txt file in the current directory.

As with hard links, you can omit the link name to create a link with the same name as the target in the current directory.

Symbolic links point to a path in the file system. You can create symbolic links to files and directories, even on different partitions. It is quite easy to detect a symbolic link with the output of the ls command:

$ ls -lh
total 112K
-rw-r--r-- 1 user student 110K Jun  7 10:13 target.txt
lrwxrwxrwx 1 user student   12 Jun  7 10:14 symbolicLink -> target.txt

In the example above, the first character of the permissions for the symbolicLink file is l, indicating a symbolic link. Additionally, right after the file name, the name of the target to which the link points, the target.txt file, appears.

It should be noted that in file and directory listings, soft links always show rwx permissions for the user, group, and others, but in practice, the access permissions for them are the same as those of the target.

Like hard links, symbolic links can be deleted using rm and moved or renamed using mv. However, special care should be taken when creating them to avoid breaking the link if it is moved from its original location.

When creating symbolic links, unless the absolute path is specified, the target's location is interpreted as relative to the link's location. This can create problems if the link or the file it points to is moved.

This is easier to understand with an example. Suppose there is a file called original.txt in the current directory and you want to create a symbolic link called softlink:

$ ln -s original.txt softlink

Apparently, everything would be fine. Let's check with ls:

$ ls -lh
total 112K
-r--r--r-- 1 user student 110K Jun  7 10:13 original.txt
lrwxrwxrwx 1 user student   12 Jun  7 19:23 softlink -> original.txt

You can see that softlink points to (->) original.txt. However, let's see what happens if you move the link to the parent directory and try to display its content:

$ mv softlink ../
$ less ../softlink
../softlink: No such file or directory

Since the path to original.txt was not specified, the system assumes it is in the same directory as the link. When this is no longer true, the link stops working.

The way to avoid this is to always specify the absolute path to the target when creating the link:

$ ln -s /home/user/Documents/original.txt softlink

This way, no matter where the link is moved, it will continue to work because it points to the absolute path of the target. This can be verified with ls:

$ ls -lh
total 112K
lrwxrwxrwx 1 user student   40 Jun  7 19:34 softlink -> /home/user/Documents/original.txt

Archiving and Compressing Files¶

'tar' Command¶

The tar command, short for “Tape ARchive”, is used to create tar files that bundle a group of files into a single file. The advantage of creating a single file is that it can be easily attached, grouped, and moved.

tar also has the ability to extract tar files, list the files included in the archive, and add additional files to an existing archive.

The syntax of the tar command is as follows:

tar [OPERATION] [OPTIONS] -f TAR_FILE [FILES...]

The basic operations that can be performed are:

  • Create -c or --create
  • Extract -x or --extract
  • List -t or --list

The most commonly used options are:

  • -f TAR_FILE or --file=TAR_FILE sets the name of the tar file.
  • -z or --gzip uses gzip compression.
  • -j or --bzip2 uses bzip2 compression.
  • -J or --xz uses xz compression.
  • -v or --verbose shows the files as they are processed.
  • -C DIRECTORY or --directory=DIRECTORY changes to the specified directory before processing the files.

Here are some examples of how to use the tar command:

  • Create a tar archive of a directory:
$ tar -cvf archive.tar Desktop/

In this case, an uncompressed archive.tar is created with all the contents of the Desktop directory.

  • Extract a tar archive:
$ tar -xvf archive.tar

This will extract the contents of archive.tar into the current working directory.

  • Create a compressed tar archive in gzip (tar.gz):
$ tar -czvf documents.tar.gz Documents/*.txt Documents/*.odt

In this example, the z option is added to compress the result in gzip. You can see that a list of files is passed using a relative path and wildcards.

  • Decompress a tar archive compressed in gzip (tar.gz):
$ tar -xzvf documents.tar.gz -C som/ud9

- The option is changed to **`x`** to extract the contents.
- The **`-C`** option is added to specify the path of the destination directory where the files will be decompressed. If `-C som/ud9` is not specified, the files will be decompressed in the current working directory.
  • Create a compressed tar archive in bzip2 (tar.bz2):
$ tar -cjvf photos.tar.bz2 *.jpg

The main feature of bzip2 is that it achieves a higher compression ratio, although it is somewhat slower than gzip.

  • List the contents of a tar archive:
$ tar -tvf photos.tar.bz2

This would display a list of the contents of the tar archive without extracting or altering the contents.

Searching for Files and Directories 'find'¶

To search for files on a Linux system, we can use the find command. This is a very powerful tool, full of parameters that can be adapted and modify the output according to our needs.

find needs two arguments:

  • starting point
  • what to search for

For example, to search for all files in the current directory (and subdirectories) whose names end in .jpg, you can use:

$ find . -name '*.jpg'
./holidays2024.jpg
./Mint21.jpg
./Forest.jpg
./Beach.jpg
./Mountain.jpg
./Mountain2.jpg
./Room.jpg
./Tolkien.jpg

This will match any file whose last four characters in the name are .jpg, regardless of what comes before, as * is a wildcard for “anything”.

The -name option is case-sensitive. If you want to perform a case-insensitive search, you can use -iname.

By default, find will start at the starting point and descend through any subdirectories it finds recursively. You can restrict this behavior with the -maxdepth N parameter, where N is the maximum number of levels.

To search only in the current directory, use -maxdepth 1. Assuming the following directory structure:

directory/                          <-- maxdepth 1 (current directory)
├── clients.txt
├── partners.txt -> clients.txt
└── somedir/                        <-- maxdepth 2
  ├── anotherdir/                   <-- maxdepth 3
  └── clients.txt

To search within somedir, you would need to use -maxdepth 2 (the current directory +1 level down). To search within anotherdir, you would need -maxdepth 3 (the current directory +2 levels down). The -mindepth N parameter works in the opposite way, searching only in directories at least N levels down.

The -mount parameter can be used to prevent find from descending into filesystems mounted within the hierarchy. You can also restrict the search to specific types of filesystems using the -fstype parameter. So find /mnt -fstype exfat -iname "*SOM*" would only search within exFAT filesystems mounted on /mnt.

Searching by Attributes¶

The following parameters can be used to search for files with specific attributes, such as those writable by the user, having a specific set of permissions, or having a certain size:

  • -type TYPE

Searches only among files of the specified type, where TYPE can be one of the following values:

  • d directory
  • f file
  • l symbolic link

For example, to search in the ./Documents directory for files containing doc in their name, you can use:

$ find ./Documents -type f -name '*doc*'
  • -user USERNAME

Matches files owned by the user USERNAME.

  • -group GROUPNAME

Matches files owned by the group GROUPNAME.

  • -readable

Matches files that are readable by the current user.

  • -writable

Matches files that the current user can write to.

  • -executable

Searches for files that are executable by the current user. In the case of directories, this will match any directory that the user can enter.

  • -perm NNNN

This will match any file that has exactly the permission NNNN. For example, -perm 0664 will match any file that the user and group can read and write and that others can read (or rw-rw-r--).

You can add a - before NNNN to search for files that have at least the specified permission. For example, -perm -644 would match files that have at least 644 permissions (rw-r--r--). This includes a file with 664 (rw-rw-r--) or even 775 (rwxrwxr-x).

  • -empty

Matches empty files and directories.

  • -size N

Matches any file of size N. The following suffixes can be used:

  • c - bytes (characters)
  • k - Kilobytes
  • M - Megabytes
  • G - Gigabytes

You can add the prefixes + or -, meaning larger than or smaller than, to search for relative sizes. For example, -size -10M will match any file smaller than 10 MB.

Another example, to search for files in the home directory that contain the pattern som case-insensitively anywhere in the name, have 0644 permissions, were accessed 10 days ago, and are at least 1 MB in size, we can use:

$ find ~ -iname "*som*" -perm 0644 -atime 10 -size +1M

Searching by Timestamps¶

In addition to searching by attributes, we can also perform searches by time, finding files that were accessed, had their attributes changed, or were modified during a specific time period.

We need to understand a file's timestamps can be a bit tricky. There are three timestamps associated with a file:

  • Access time (atime): The last time the file was accessed (read).
  • Modification time (mtime): The last time the file was modified (written to).
  • Change time (ctime): The last time the file's metadata was changed (permissions, ownership, etc.).

stat

The stat command can be used to view the timestamps of a file. For example, to view the timestamps of a file named file.txt, you can use:

$ stat file.txt
  File: file.txt
  Size: 12              Blocks: 8          IO Block: 4096   regular file
Device: 0,51   Inode: 4513695     Links: 1
Access: (0644/-rw-r--r--)  Uid: ( 1000/ user)   Gid: ( 1000/ user)
Context: unconfined_u:object_r:user_home_t:s0
Access: 2025-03-28 08:55:17.324937575 +0100
Modify: 2025-03-14 10:34:39.755854244 +0100
Change: 2025-03-14 10:34:39.755854244 +0100
Birth: 2025-03-14 10:32:24.620427666 +0100

The following parameters can be used:

  • -amin N, -cmin N, -mmin N

This will match files that were accessed, had their attributes changed, or were modified (respectively) N minutes ago.

  • -atime N, -ctime N, -mtime N

This will match files that were accessed, had their attributes changed, or were modified N*24 hours ago.

For -cmin N and -ctime N, any attribute change will trigger a match, including a change in permissions, reading, or writing to the file. This makes these parameters especially powerful, as practically any operation involving the file will trigger a match.

The following example would match any file in the current directory that was modified less than 24 hours ago and is larger than 100 MB:

$ find . -mtime -1 -size +100M

Executing Commands on Found Files¶

The find command can also execute commands on the files it finds. This is done using the -exec parameter, followed by the command to execute and {} to represent the file found. The command must end with a semicolon ;.

For example, to delete all files in the current directory that are older than 30 days, you can use:

$ find . -mtime +30 -exec rm {} \;

The rm command is executed for each file found, and {} is replaced by the file name. The \; is used to indicate the end of the command.

Combining Conditions¶

You can combine conditions using -and, -or, and -not. For example, to search for files that are not writable by the user and are larger than 1 MB, you can use:

$ find . -not -writable -and -size +1M

Another example, to search for files that are either more than 10MB or have been modified in the last 10 days, you can use:

$ find . -size +10M -or -mtime -10

A complex example, to search and backup in /backup directory in targ.gz format, all files in the ~/Documents directory that are writable by the user and are larger than 1 MB or modified in the last 7 days, you can use:

$ find ~/Documents -writable -and \( -size +1M -or -mtime -7 \) -exec tar -czvf /backup/backup.tar.gz {} \;

Parentheses

Parentheses are used to group conditions. The backslash \ is used to escape the parentheses, as they are special characters in the shell.
